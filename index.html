The Vision: EVM on a Privacy Blockchain

The Fundamental Question (2022-2023)

Problem

Can Ethereum code run on privacy blockchain?

→

Research

18 months of focused research and experimentation

→

Answer

YES Feasible at Proof-of-Concept level

The answer was yes. In May 2023, the team published the first proof of concept for EVM on CryptoNote.

The Proof

Proof of Concept Achievement (May-July 2023)

Research Paper

  • Published: May 5, 2023
  • Authors: Allard Dijk (BitBender) & Dr. Dominique Schröder (University of Erlangen-Nuremberg)
  • Title: "Proof of Concept for EVM on CryptoNote"
  • Achievement: First EVM on CryptoNote blockchain
  • Significance: Groundbreaking blockchain research

Testing

  • Network: 16 synchronized PoS nodes
  • Implementation: Minimized Py-EVM (Python-based)
  • Tests: Smart contract creation, deployment, execution
  • Validation: Transaction triggering and state management
  • Result: All core functionality proven feasible

What Was Proven

  • Privacy Maintained: RingCT still functional on Layer 1
  • Full EVM Compatibility: Can run Ethereum smart contracts
  • State Management: UTXO ↔ Account bridge possible
  • Production Path: Clear roadmap identified

With feasibility proven, the next question became: what does this system actually look like?

The Technical Challenge

UTXO → Account (SOLVED)

Users burn coins on main chain, receive credit on EVM layer

Challenge: Prevent double-spending of burn proofs while maintaining privacy

🔄 Account → UTXO (IN RESEARCH)

Moving funds back to private main chain

Status: Still under active research.

This hybrid approach combines the strengths of two fundamentally different models.

UTXO vs Account Model Comparison

Feature UTXO Model (Bitcoin/Monero) Account Model (Ethereum) Beldex Hybrid
Structure Unspent transaction outputs Account balances Uses UTXO for transfers and account-style state for smart contracts
Privacy ✅ Strong privacy (RingCT) ❌ All public ✅ Privacy maintained
Smart Contracts ❌ Very limited ✅ Full support ✅ EVM-compatible (research PoC)
Parallel Processing ✅ Yes ❌ Sequential ✅ Yes (Layer 1)
State Management ❌ Stateless ✅ Global state ✅ Both approaches

Here is how a smart contract moves through this architecture, step by step.

How Smart Contracts Work on Beldex (Proven in PoC)

Step 1: Contract Creation

Developer writes Solidity smart contract (exactly like Ethereum)

Tools: Remix, Hardhat, Truffle - all compatible

Step 2: Transaction Submission

User creates Beldex transaction with contract bytecode embedded

Private transaction amounts via RingCT on Layer 1

Step 3: Block Inclusion

Contract transaction added to block by PoS validator

Validated through masternode network consensus

Step 4: EVM Execution

Python-based Py-EVM processes contract on all synchronized nodes

Contract code is public, execution verified by all

Step 5: State Update

Contract state updated globally, subsequent transactions can trigger functions

Persistent state across the network

Step 6: Privacy Preservation

Contract code is PUBLIC (anyone can verify)

Transaction amounts & identities remain PRIVATE (RingCT)

Future: ZK-SNARKs for contract-level privacy

Step 6 raises a critical question: how is privacy maintained when contract code must be public?

Privacy Strategy Explained

❌ REJECTED

Encrypted Smart Contracts

Idea:

Encrypt the smart contract code itself

Critical Problem:

Validators cannot execute or verify encrypted code

Why It Fails:

Decentralized networks require public verification for consensus

Fundamentally impossible ✗
CHOSEN

Layer-Based Privacy

Layer 1 (Main Chain):

RingCT hides amounts, senders, and receivers

Smart Contract Layer:

Code is public, but transaction data remains private

Future Enhancement:

ZK-SNARKs add privacy to contract inputs/outputs

Transparent + Private ✓

Three-Tier Privacy Model

👁️

Tier 1: Public

Contract Code

Like open-source software - everyone can verify the logic and execution

🔒

Tier 2: Private

Transaction Data

Amounts, balances, and user identities - all encrypted and hidden

🔐

Tier 3: Protected

Financial Privacy

ZK-SNARKs prove validity without revealing underlying data

Building this system means solving problems on blockchain has faced before. Here is where each challenge stands.

Technical Challenges & Solutions

Core engineering problems Beldex is solving to enable privacy-preserving smart contracts

01

UTXO vs Account Incompatibility

These two systems work in very different ways, so they don't fit together easily.

✓ PoC Proven
PROBLEM

Fundamental Conflict

  • Beldex: Uses UTXO → best for privacy
  • Ethereum: Uses accounts → needed for smart contracts
  • Issue: These two systems work in very different ways, so they don't fit together easily.
SOLVED

Hybrid System

  • Approach: Use a hybrid system (combine both styles).
  • How it works: Use UTXO-style transactions for privacy, and accounts-style transactions for smart contracts.
  • Benefit: Privacy and smart contracts can coexist.
02

Double Spending Prevention

Smart contracts don't use keys, so value locked by a contract could be used twice without a fix.

✓ PoC Proven
PROBLEM

Value Reuse Risk

  • Challenge: Smart contracts don't use keys
  • Risk: Value locked by contract used twice
  • Core Issue: One-time usage without breaking privacy
SOLUTION (PoC)

Cryptographic Linking

  • Approach: Protocol-level consensus rules
  • How it works: Unique state transitions linked
  • Benefit: Reuse prevented + privacy maintained
03

Two-Way Bridge Architecture

Moving value between the privacy chain and EVM layer needs to work in both directions, one direction is solved, one is still being researched.

⟳ Half Solved
SOLVED

UTXO → Account

  • Method: Burn coins on main chain
  • Proof: Cryptographic burn proof
  • Result: Credit on contract layer
IN RESEARCH

Account → UTXO

  • Challenge: Valid RingCT generation
  • Research: Proof of withdrawal
  • Status: Active investigation ongoing
04

Smart Contract Secrecy

Contracts need to see data to execute, but privacy requires all data to be hidden, a fundamental paradox.

⟳ In Research
PROBLEM

The Core Paradox

  • Contracts: Need to see data to execute
  • Privacy: All data must be hidden
  • Paradox: How to execute without seeing?
SOLUTION

Layered Privacy

  • Code: Public (verifiable by all nodes)
  • Data: Private via RingCT on Layer 1
  • Future: ZK-SNARKs for contract privacy
05

Blockchain Size & Scalability

Large privacy proofs caused slow sync times and inflated storage, now solved with Bulletproof++ compression.

Solved
PROBLEM

Transaction Bloat

  • Before: 1.2–4 MB per 100 tx
  • Impact: Slow sync, large storage
  • Limitation: Scalability bottleneck
SOLVED

Bulletproof++ Compression

  • Technology: Bulletproof++ ZK-proofs
  • After: ~120–400 kB per 100 tx
  • Status: Deployed Dec 7, 2025
06

Network Validator Security

Attackers can predict which validators will sign the next block, VRF randomisation will make that impossible.

◎ Planned
PROBLEM

Predictable Selection

  • Current: 11+1 deterministic validators
  • Risk: Attackers can predict targets
  • Issue: Targeted attack vulnerability
FUTURE

VRF Randomization

  • Selection: Verifiable random functions
  • Participation: ~33% of all nodes
  • Security: Must attack 1/3 of network
