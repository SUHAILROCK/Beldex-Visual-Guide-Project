Section 01

What exactly is BelNet?

BelNet is a decentralized VPN but that description alone misses what makes it different. A normal VPN replaces one problem (your ISP sees everything) with a different problem (the VPN company sees everything). BelNet eliminates the problem entirely by removing any central server from the equation.

Switch between the two views below to see exactly what your internet traffic looks like with and without BelNet running.

Why "decentralized" matters

A centralized VPN is like a security guard you hire they can still read everything you give them. If the guard is corrupt, forced by law, or hacked, your data is exposed. BelNet is different in three fundamental ways:

No central server. Your traffic is routed through 2200+ independent community-run masternodes. There is no single server to hack, seize, or compel to hand over data.
No company to pressure. There is no BelNet corporation that can receive a court order or be pressured into revealing user data.
Zero logs by architecture. The LLARP protocol does not log connection data. Not by policy by design. There is nothing to hand over because nothing is stored.
The key difference from Tor

Tor runs on volunteer nodes with no economic incentive to stay online or behave honestly. BelNet nodes are economically staked every node operator has locked 10,000 BDX as collateral. Misbehaving nodes can be penalized. This creates a fundamentally more reliable and trustworthy network.

Section 03

The LLARP Protocol

LLARP stands for Low Latency Anonymous Routing Protocol. It is the technical heart of BelNet the rulebook that says exactly how data should be encrypted, wrapped, and sent through the network. Think of it as the postal system's delivery rules, except every package is sealed in multiple envelopes, and each delivery driver only knows the next stop.

Why LLARP uses UDP, not TCP

Most internet protocols use TCP, which adds "delivery confirmation" to every packet. This confirmation adds latency. For a VPN that already adds multiple relay hops, TCP latency compounds making it unpleasantly slow for real-time apps like video calls.

LLARP uses UDP it sends packets without waiting for confirmation, achieving much lower latency. Watch the race below. Press "Start Race" to see the difference.

What makes LLARP unique

Router List from Blockchain
Tor relies on centralized "directory authorities" to tell nodes who else is on the network. LLARP gets this list from the Beldex blockchain fully decentralized, no single point of failure.
Built-in NAT Traversal
Works on mobile devices, behind routers with CGNAT, without any port forwarding or configuration. Most VPN protocols fail in these environments.
Path Rotation Every 10 Minutes
Every 10 minutes, LLARP automatically builds a new circuit through different nodes. This prevents traffic analysis attacks where someone might try to correlate your long-term patterns.

Onion Routing Layer by Layer

The name "onion routing" comes from the idea of wrapping your message in multiple layers of encryption like an onion. Each relay node in the network can only peel one layer. After peeling, it sees only the address of the next relay and nothing else. It then passes the remaining layers forward. Click each ring below to peel it and see exactly what each relay learns.

Exit Nodes Your Internet Gateway

After traveling through three encrypted hops, your traffic needs to exit into the regular internet. This is done by a specialized masternode called an exit node. The exit node forwards your request to the destination website using its own IP address, not yours. The website sees only the exit node.

What if an exit node is compromised?

Even if an attacker controls an exit node, they learn only the destination website not who sent the request. They cannot trace backwards through the circuit because Bhava (middle relay) only told them the traffic came from Jothi (entry relay). Without also controlling Jothi, the attacker knows nothing about the sender.

Section 07

BNS The .bdx Domain System

The Beldex Name System (BNS) is a naming layer built on the Beldex blockchain. Just like DNS translates "google.com" into an IP address, BNS translates a short human-readable name like kisore.bdx into a BelNet service address or a BChat ID. The key difference: BNS is fully on-chain. There is no registrar to comply with takedown orders.

Why on-chain DNS matters

Regular DNS is censored everywhere
Governments regularly order domain registrars to "unpublish" websites. The registrar complies, and suddenly example.com no longer resolves. Your website is gone with no recourse.
BNS registration is permanent
When Bhava registers bhava.bdx on the Beldex blockchain, that record is written permanently. No authority can remove it only Bhava's private key can modify it.
One name, multiple uses
The same .bdx name links to both a BelNet service address (for website hosting) and a Beldex wallet address (for payments). Friends can message and pay using just one name.

Section 09

Split Tunneling Your Traffic, Your Rules

BelNet's split tunneling works at the application level not the network level. This means you choose exactly which apps route through the encrypted BelNet tunnel and which connect directly to the internet. Sensitive apps (banking, browser) get full privacy. Bandwidth-heavy apps (streaming, gaming) connect directly at full speed.

Application-level vs Network-level

Most VPN split tunneling works by modifying your device's routing table routing some IP ranges through the VPN and others directly. This can leak traffic if rules are misconfigured. BelNet intercepts at the socket level each app's connections are individually handled before they ever touch the routing table.

Result: zero leak risk

Because the decision happens at the app level (not the IP level), there is no routing table gap that traffic can accidentally fall through. Direct apps are simply never handed to the VPN stack.

The Security Model

No privacy system is perfect. The right question is: which attacks are realistic, and what resources do they require? BelNet is designed so that all practical attacks either require an implausibly large amount of resources, or are mathematically impossible by design. Click each threat to see the analysis.

Why 3 hops is the right number

With 2 hops: if an adversary controls both the entry and exit node, they can link sender to destination. With 3 hops: even controlling both entry and exit still leaves the middle relay unknown. An attacker would need to control all three simultaneously across 2200+ randomly selected nodes which requires enormous capital (at 10,000 BDX per node) and is statistically near-impossible.
