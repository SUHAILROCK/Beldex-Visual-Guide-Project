STAGE 01 / 09
Wallet Layer
OUTPUT
DETECTION

Before you can spend BDX, your wallet must find what you own. It does this by scanning every block and checking each output against your private view key. This process is completely silent. Nobody on the network knows your wallet is scanning.

Every output on the blockchain is public but the recipient is not. Each output uses a one-time stealth address.
Your private view key can test any output to see if it belongs to you. It cannot spend only detect.
When a match is found, the output is marked as a UTXO (Unspent Transaction Output) and added to your balance.
Your wallet always knows your balance because it tracks all owned outputs and which ones have been spent.
Key Fact
View Key = Read Only
Your private view key can scan outputs. Only your private spend key can authorize a transaction.

STAGE 02 / 09
Crypto Layer
RING
ASSEMBLY

To hide which output you are spending, your wallet mixes your real input with 10 decoy outputs pulled from the blockchain. This group is called a ring. Everyone on the network sees all 11 candidates, but cannot tell which one is the true spend.

Ring size is 11 on Beldex. 1 real input. 10 decoys. Decoys are real past outputs they just are not being spent right now.
Decoys are selected by your wallet using a statistical distribution to mimic natural spending patterns.
A Key Image is generated from your real output. It is unique per output. It prevents double-spending without revealing which ring member you spent.
A ring signature is produced proving that someone in the ring authorized this transaction. Mathematically it is impossible to determine who.

STAGE 03 / 09
Wallet Layer
TRANSACTION
CONSTRUCTION

With the ring assembled and the destination known, your wallet builds the full transaction object. Think of this as packing a sealed envelope with everything the network needs minus any identifying information.

version tells the network which protocol rules apply. Beldex is currently on version 20 (HF20 Obscura).
inputs reference the ring of 11 outputs plus the key image. They do not reveal the real spend.
outputs are addressed to the receiver's stealth address. Change goes back to you via a second stealth output.
rct_signatures holds the RingCT data and Bulletproofs+ proof. This is the largest part of the transaction by byte size.
unlock_time is typically 0 (spend immediately). It can be set to a future block height to time-lock an output.

STAGE 04 / 09
Crypto Layer
RINGCT
SEALING

RingCT stands for Ring Confidential Transactions. It hides every amount in the transaction using a technique called a Pedersen Commitment. No validator, no node, and no observer can read how much BDX was sent.

A Pedersen Commitment is a mathematical "locked box" around a number. You can prove two boxes balance without opening either one.
The formula is C = xG + aH where a is the real amount, x is a random blinding factor, G and H are elliptic curve points. The amount is invisible.
Balance is proven because: Sum(C_inputs) = Sum(C_outputs) + C_fee. The equation holds even though no amounts are visible.
ECDH encryption is used to share the real amount privately with the receiver only. Nobody else can decrypt it.

STAGE 05 / 09
Crypto Layer
BULLETPROOFS+
RANGE PROOF

RingCT hides amounts. But how do validators know you are not creating BDX from nothing? Bulletproofs+ is the answer. It generates a tiny proof that certifies every output amount is between 0 and 2^64 without revealing the actual value.

A range proof prevents inflation attacks. Without it, someone could create a commitment to -1 BDX and trick the balance equation into passing.
Bulletproofs+ (HF20) reduced proof size by 38% compared to standard Bulletproofs. One proof for the entire transaction.
Verification is fast because Bulletproofs+ supports batch verification. Many proofs can be checked simultaneously.
The proof size history shows continuous improvement: Borromean 6.1 KB to Bulletproofs 675 B to Bulletproofs+ 417 B. A 93% total reduction.

STAGE 06 / 09
Crypto Layer
STEALTH
DELIVERY

Even if someone knows your wallet address (your public key), they still cannot trace any payment to you. That is because every transaction generates a completely new one-time address just for that payment. This is called a stealth address.

The sender uses your public spend key and public view key combined with a random scalar r to generate a unique output key P.
The key P appears on the blockchain as the output target. Nobody knows it belongs to you it looks like a random key.
You scan with your private view key and find outputs addressed to you. Only you can compute the matching private key to spend them.
The address is never reused. Even two payments to the same wallet look completely unrelated on-chain.
ECDH Stealth Key Derivation
01
SENDER: Generate Random Scalar
Sender picks a random number r. This is kept secret. It seeds the entire key derivation.
r = random_scalar()   // only sender knows this
02
DERIVE: One-Time Output Key
The sender computes a shared secret using ECDH (Elliptic Curve Diffie-Hellman) with the receiver's public view key V.
P = H(r·V) · G + B   // B = receiver public spend key
03
SENDER: Publish Transaction Key
The value r*G (called the transaction public key R) is embedded in the transaction. It lets the receiver derive the same shared secret.
R = r·G   // published in transaction extra field
04
RECEIVER: Scan and Detect
The receiver uses their private view key v and the published R to compute the same output key P. If P matches a blockchain output, that output belongs to them.
P_check = H(v·R) · G + B   // scan with private view key
STAGE 07 / 09
Network Layer
P2P
BROADCAST

Once the transaction is sealed and signed, your wallet sends it to the Beldex peer-to-peer network. The transaction immediately starts propagating from node to node until the entire network has a copy.

Your wallet connects to multiple peers and sends the raw transaction bytes to each. They forward it to their peers as well.
Each node validates on receipt: correct version, valid ring size, key image not already spent, valid signature format.
The transaction enters the mempool a waiting room for unconfirmed transactions. The block producer will pull from here.
Higher fee transactions are generally prioritized. On Beldex, fees are extremely low due to compact transaction sizes.
Network Fact
Propagation: < 2 Seconds
A transaction reaches most of the Beldex network within 1 to 2 seconds of broadcast.

STAGE 08 / 09
Consensus Layer
QUORUM
CONSENSUS

Beldex does not use miners. It uses a proof-of-stake system where 12 masternodes are randomly selected every block. One builds the block template. The other 11 validate it. 7 signatures out of 11 are required to approve the block.

Selection is verifiable random. No masternode operator can predict or manipulate which quorum they will be in.
The block producer assembles transactions from the mempool, generates the block header, and sends it to the other 11 validators.
Validators check every transaction: valid ring signatures, key images not spent, valid Bulletproofs+, correct fee, valid RingCT commitments.
7 / 11 threshold means the network tolerates up to 4 corrupt or offline validators in any quorum round.
Block time is 30 seconds. If a quorum fails, a new one is selected and the process restarts.

STAGE 09 / 09
Chain Layer
CHAIN
FINALITY

Once the quorum approves the block, it is added to the Beldex blockchain. Three things happen immediately: the key image is recorded (preventing double-spends), the receiver can detect their new output, and the chain checkpoint system begins locking the transaction in permanently.

Key image recording is the double-spend prevention mechanism. The same output can never be spent twice on the network.
The receiver scans the new block using their private view key and detects the stealth output addressed to them.
10-block confirmation is the practical irreversibility threshold. After 10 blocks (~5 minutes), reorgs that could affect the transaction are essentially impossible.
Checkpoint blocks are committed every 4 blocks by quorum consensus. They permanently lock all prior history against any rollback attack.
