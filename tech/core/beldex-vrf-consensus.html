Section 01

What Problem
Does VRF Solve?

Before we talk about VRF, we need to understand why picking a block producer is so hard. It sounds simple, but it opens the door to cheating if done wrong.

🎯
Problem 1: Predictable Selection
If everyone knows who will produce the next block before it happens, attackers can flood that node with traffic and shut it down. The network grinds to a halt.

Analogy: Imagine if the lottery winner was announced two hours before the draw. Criminals could target that person in advance.
🎲
Problem 2: Manipulated Randomness
If the random source can be influenced, a powerful node operator can keep re-rolling until a favorable result comes up. The selection is no longer fair.

Analogy: A loaded dice in a board game always lands on 6. The randomness is fake.
🤷
Problem 3: Unverifiable Claims
If a node claims "I was randomly selected," other nodes need a way to confirm that. Without proof, dishonest nodes can lie about their selection.

Analogy: Saying "I won the lottery" without showing the ticket. Nobody can confirm or deny it.
Without VRF: Vulnerable
✗

✗

✗

With VRF: Protected

Section 02

How VRF Actually Works

VRF stands for Verifiable Random Function. It takes two inputs and produces two outputs. The inputs are a public seed and a private key. The outputs are a random-looking number and a proof that the number is genuine.

1
Compute the Score

When a new block needs to be added, every active masternode picks up two pieces of information: the hash of the previous block (public : everyone can see it) and their own private key (secret : only they know it). They feed both into the VRF function and get back a unique random number called their VRF score and a cryptographic proof.

Simple Analogy
Think of it like everyone drawing a number from a hat, but the hat is sealed and the number drawn is determined by who you are (private key) and what round it is (block hash). Different person, different number. Same person, same number every time for that specific round.
// Every masternode runs this locally, privately function computeScore(prevBlockHash, myPrivateKey) { // prevBlockHash is public, everyone sees the same seed // myPrivateKey is secret, only I know mine score = vrfFunction(prevBlockHash, myPrivateKey) proof = generateProof(myPrivateKey, score) return { score, proof } // one unique pair per node per block }
2
Broadcast the Proof

Each masternode broadcasts its score and proof to the entire Beldex network. The proof acts like a signed certificate : it says "this score was correctly computed using my private key" without actually revealing what that private key is. Nobody can fake this.

Simple Analogy
Like a sealed envelope with your score inside and a signature on the outside. Anyone can verify the signature is yours, but nobody can change the score inside without breaking the seal.
// Broadcast to all nodes in the Beldex network network.broadcast({ myPublicKey: "bdmn_1a9f3c...", // who am I vrfScore: myScore, // my random number vrfProof: myProof, // proof I computed it correctly blockSeed: "prev_block_hash" // which block this is for })
3
Verify, Rank and Select

Every node receives all the scores and proofs from the network. Using each masternode public key, they verify every single proof. No trusted middleman is needed. Then all verified scores are sorted from lowest to highest. The masternode with the lowest score wins : they become the block producer. The next 11 lowest scores become the quorum validators.

Simple Analogy
Imagine everyone drew a number and showed it publicly with proof it was genuine. The person who drew the lowest number builds the block. The next 11 people double-check the block to make sure it is valid.
// Every node independently runs this same logic let verifiedScores = [] for (each received broadcast) { if (verifyProof(publicKey, blockSeed, score, proof)) { verifiedScores.push({ node, score }) } } verifiedScores.sort((a, b) => a.score - b.score) // Result is the same on every single node blockProducer = verifiedScores[0] // lowest wins quorumMembers = verifiedScores[1..11] // next 11 validate

Section 04

The Quorum

Selecting a block producer is only half the job. A single node producing a block alone could make mistakes or act dishonestly. That is why Beldex uses a quorum : a committee of 12 nodes that work together every round.

1
Block Producer: The masternode with the lowest VRF score. It collects pending transactions and assembles them into a block template. It then proposes this block to the validators.
11
Validators: The next 11 lowest scores. Each validator independently checks every transaction in the proposed block. They verify signatures, balances, and privacy proofs.
7
Minimum Signatures Required: At least 7 of the 11 validators must sign the block before it is accepted. This is called a supermajority. Up to 4 validators can be offline or dishonest and the block still gets approved.
0
Overlap with Previous Quorum: Each quorum is freshly selected using the new block VRF scores. The committee changes every single block. No node can build influence over multiple consecutive rounds.

Section 06

Security Guarantees

VRF does not just provide randomness. It comes with formal mathematical guarantees that protect the entire network. Here is what each property means in plain terms.

🎲
Property 01
Unpredictability
"Nobody knows who wins until it happens."
Before a masternode publishes its score, nobody, not even the masternode itself : can know how the scores will rank. The result is computationally indistinguishable from random. Attackers cannot pre-target the incoming block producer.
🔒
Property 02
Uniqueness
"One key, one score. No re-rolling."
For any given seed and private key, there is exactly one valid VRF output. A node cannot compute multiple valid scores and pick the best one. The function is one-to-one : one key always produces the same score for the same block.
✅
Property 03
Verifiability
"Everyone can check everything."
Any node can verify any VRF score using only the public key. Verification is instant, requires no interaction with the prover, and always gives the same answer. There is no trusted authority and no need for anyone to take anyone's word for it.
🛡️
Property 04
Tamper-Proof
"You cannot fake someone else's score."
An attacker cannot construct a fake VRF output that passes verification. Doing so would require breaking the elliptic curve cryptography that secures the entire internet. With current computing technology, this is not possible.
🚫
Property 05
DoS Resistance
"Attackers have no target until it is too late."
The winning node is not revealed until the previous block is finalized. At that point, it is already time to build the next block. By the time an attacker tries to interfere, the block is already being built. The window for attack is essentially zero.
⚖️
Property 06
Equal Fairness
"Every masternode has the same chance."
All masternodes lock the same collateral (10,000 BDX). The VRF output is uniformly distributed across all nodes over time. Over many blocks, each node wins approximately the same number of times : regardless of how long it has been running.
