HF1 - HF3
Genesis Configuration
CryptoNote base layer. Ring sigs, stealth addresses, RingCT enforced from block one. Three early forks calibrate emission and block timing.
▶
Date
2018
Consensus
CryptoNight-Heavy PoW
Base Protocol
CryptoNote (Monero fork)
  • Ring Signatures mandatory - Every transaction input is mixed with decoy outputs. Sender identity is hidden by default. No transparent mode exists.
  • Stealth Addresses - The sender generates a one-time address per payment using ECDH. The recipient's real address never appears on-chain.
  • RingCT enforced on all transactions - Transaction amounts hidden using Pedersen commitments where C = xG + aH. No amount is visible to external observers.
  • Key images I = x*Hp(P) - Each spend produces a unique key image that prevents double-spending without revealing what was spent.
  • Emission curve calibrated - Block reward decay schedule set for long-term supply sustainability.
  • HF1 through HF3 - Three rapid stabilization forks in the first weeks tune emission and network parameters.

Ring Size Expansion
Minimum ring size increased. Larger anonymity set per input. Statistical tracing attacks become exponentially harder.
▶

Change Type
Privacy Hardening
Effect
Larger anonymity set
  • Minimum ring size increased - More decoy outputs are required per real input in every transaction. The anonymity set grows exponentially.
  • EAE attack resistance improved - Even-after-even timing analysis becomes statistically negligible. Tracing probability drops toward theoretical noise floor.
  • Mandatory wallet enforcement - Wallet software is required to submit the new minimum ring size. Smaller rings are rejected at the protocol level.
  • No opt-out - Unlike optional privacy in some chains, the ring size increase applies to every transaction without exception.

Masternodes launch as a second layer alongside PoW miners. 10,000 BDX collateral required. Reward queue introduced.
▶

Collateral

Model
Hybrid PoW + MN
  • 10,000 BDX collateral required - Operators lock BDX to register a masternode. Staked coins are returned on deregistration.
  • Hybrid model active - PoW miners build blocks. Masternodes validate them and share the block reward. Both layers operate simultaneously.
  • Reward queue system - Masternodes receive rewards in round-robin order. A node that just received a reward moves to the back of the queue. Ensures fair distribution across all operators.
  • Deregistration logic - Masternodes that go offline or misbehave are deregistered. Staked BDX is returned but queue position is lost.
  • Foundation for PoS transition - The masternode layer built here will eventually replace mining entirely in Bucephalus (HF17).

Proof Reduction

Size Reduction
89% smaller
  • Borromean range proofs retired - Old proofs scaled linearly. A 2-output transaction required approximately 6.1 KB of range proof data alone.
  • Bulletproofs activated - New logarithmic-scaling proofs reduce a 2-output transaction proof to 675 bytes. An 89% size reduction in one upgrade.
  • Batch verification enabled - Multiple Bulletproofs can be verified together in a single pass. Masternode validation throughput increases significantly.
  • Transaction fees drop - Smaller transactions cost less. Privacy on Beldex becomes cheaper to use at every volume level.
  • Same security guarantees - Bulletproofs maintain full zero-knowledge range proof security. No trusted setup required.

Pre-PoS Refinements
2020 to 2021
Three rapid hardforks prepare the architecture for Proof of Stake. The masternode reward system is hardened. The dApp infrastructure is tested on testnet before mainnet activation.
  • Batched governance outputs replace per-block payments. Thousands of tiny UTXOs are eliminated from the set. Node sync time improves measurably.
  • Masternode reward queue formalized across all client implementations. Consensus on queue state is reached deterministically by all nodes.
  • Storage server integration tested on testnet. BChat message routing infrastructure is embedded in masternode software for the first time.
  • BelNet service node routing begins integration testing. The decentralized VPN layer is being built into the masternode network.
13
Awakening
Batched Governance
2021
Governance reward payments are fully batched. Instead of per-block outputs, governance funds accumulate over many blocks and are distributed in one transaction. The UTXO set growth rate slows significantly.
  • Periodic batch payouts replace constant small governance outputs. A large batch every N blocks instead of a small one every block.
  • UTXO set growth reduced. Fewer outputs means faster full node sync and smaller storage requirements for all node operators.
  • Chain overhead reduced. Less governance data per block leaves more space available for user transactions.
14
Awakening
Checkpoint Enforcement
2021
Masternodes begin signing checkpoint hashes at regular intervals. Once a block depth passes the threshold, reorganization becomes impossible. Finality is now mathematically guaranteed, not probabilistic.
  • Checkpoint hashes co-signed by masternode quorum every few blocks. The signed checkpoint is embedded in subsequent block data.
  • 51% attack prevention. A miner with majority hash power cannot reorganize the chain past a signed checkpoint boundary. The attack window is permanently bounded.
  • Exchange finality improved. Deposits can be safely confirmed after checkpoint confirmation rather than waiting for long reorg-safe depths.

Flash instant transactions activate on mainnet. Sub-2-second confirmation via masternode quorum consensus. The dApp infrastructure node layer comes online.
  • Flash transaction lock. A quorum of masternodes co-signs a transaction instantly. Merchants can accept BDX with under 2-second confirmation.
  • Beldex Storage Server fully integrated. Masternode operators run message routing for BChat. Privacy messaging is routed through the node network.
  • BelNet dVPN active. IP-level traffic routing through masternodes is live on mainnet. The decentralized VPN layer begins serving real users.

Miners Removed. Masternodes Rule.
The Bucephalus hardfork, activated at block 742,421 on December 10, 2021, is the single most consequential upgrade in Beldex history. With one block, the CryptoNight-Heavy mining algorithm was permanently disabled. Miners, who had secured the network since the 2018 genesis, ceded control entirely to the masternode network.

The minimum stake of 10,000 BDX to operate a masternode remains unchanged. However, the role of masternodes expanded from validation support to full block production. The Proof of Work code path is removed from the daemon codebase and never returns.
Four Times Faster. Five Times Richer.
Block time dropped from 120 seconds to 30 seconds. The network now produces 2,880 blocks per day versus 720 under Proof of Work. Each block pays 10 BDX versus the old 2 BDX reward - a five-fold increase in per-block output.

Masternode operators receive 6.25 BDX per block, representing 62.5% of the total reward. Governance receives 37.5%. For operators, passive income increased by 212.5% relative to the old model. The economics of running Beldex infrastructure changed permanently.
12-Node Quorum. 7 of 11 Threshold.
Bucephalus introduced a deterministic Proof of Stake quorum model. A seed derived from the previous block hash selects 12 masternodes per round - one block producer and eleven validators. The producer assembles the transaction template. Validators sign with Ed25519 keys. Seven of eleven signatures are required for finality.

This quorum model builds on HF14's checkpoint system. Every block has immediate near-finality. Double-spend attacks against confirmed transactions are no longer economically viable even with majority stake.

Changes in Hardfork 18 - Bern
Beldex Name System (BNS) live on mainnet
Users register human-readable .bdx names directly on-chain. "yourname.bdx" replaces a 97-character wallet address. BChat names are linked to messaging identities. Wallet names are non-expiring with a one-time fee. Each registration is an on-chain transaction - no centralized server holds the records.
Fee burning - tokenomics flips deflationary
All BNS registration fees are permanently destroyed on-chain. Within the first 24 hours, 1,500,000 BDX were burned while only 28,800 were minted. The supply trajectory reversed from inflationary to deflationary. The more BNS is used, the more deflationary the token becomes over time.
BelNet domain hosting enabled on-chain
BelNet .bdx namespaces are now registrable for 1, 2, 5, or 10 year periods. Burn requirements: 650 / 1,000 / 2,000 / 4,000 BDX. Private websites and services can be hosted over the dVPN network. Decentralized censorship-resistant web hosting is possible via BelNet for the first time.
BNS encryption scheme upgraded
Record encryption in BNS updated to a faster, more resource-efficient cryptographic scheme. Existing name holders can upgrade via the bns_update_mapping wallet command on a per-name basis. Improved security and measurably lower verification overhead for masternodes processing BNS lookups.
Masternode time synchronization
Masternodes periodically broadcast timestamp data to network peers. Operators can now diagnose clock drift without relying on external monitoring tools. Clock consistency across the global masternode network is improved, reducing block timing variance in quorum participation.

Changes in Hardfork 19 - Hermes
Ethereum address and ENS integration in BNS
BNS now stores Ethereum wallet addresses and ENS domain names alongside BDX identities. A user can link their ETH address directly to "yourname.bdx". This enables seamless cross-chain transactions and interoperability with the EVM ecosystem. It also lays the technical foundation for the LayerZero OFT bridge deployed in December 2025 - allowing BDX to move between Ethereum, BNB Smart Chain, Solana, Base, and Arbitrum at a 1:1 ratio.
Multi-masternode limit: 3 per VPS machine
Previously operators ran up to 10 masternodes per server. Hermes caps this at 3. This prevents centralization of network validation power in a small number of machines. Each masternode now has more dedicated CPU and RAM, improving the quality of BChat message routing and BelNet dVPN service delivery for end users.
Critical security vulnerabilities patched
Multiple security issues identified in a post-Bern protocol audit are fully addressed before any exploitation occurred. Network resilience is improved and the attack surface is measurably reduced. Specific vulnerability details are withheld per standard responsible disclosure practice to protect networks running older versions during the transition window.
macOS Sonoma compatibility restored
The Beldex daemon and wallet applications broken by Apple's macOS Sonoma toolchain update are fully fixed. The build system is updated for the new compiler environment. Mac-based masternode operators and desktop wallet users get consistent, stable operation restored across all major operating systems.
Network stability and P2P improvements
Underlying P2P layer improvements reduce latency between masternodes. Quorum communication timing is optimized. Block propagation speed is improved. These changes collectively reduce the occurrence of missed quorum participation events that could lead to masternode decommission warnings for otherwise healthy nodes.

Changes in Hardfork 20 - Obscura
Bulletproofs++ range proofs activated
The most advanced non-interactive zero-knowledge range proof system replaces standard Bulletproofs. Proof size for a 2-output transaction drops from approximately 675 bytes to around 417 bytes. A 38% reduction in proof data per transaction. No trusted setup is required. Full Pedersen commitment amount privacy is maintained throughout.
Higher transaction throughput per block
Beldex blocks are dynamically sized between 300 and 600 KB. Cryptographic range proofs are the dominant contributor to transaction data size in a fully private chain. Smaller proofs mean more transactions fit per block without requiring a block size increase. The network handles higher volume without hitting capacity limits.
Masternode verification overhead cut
Lighter proofs require less CPU computation and memory per validated block. Masternode hardware requirements decrease. The barrier to running a masternode falls. Network decentralization benefits as more operators can participate economically - strengthening the BChat and BelNet service layer that runs on masternode infrastructure.
Long-term chain weight problem addressed
In private blockchains, cryptographic proof data drives the majority of chain size growth - far more than transaction amounts or addresses ever do. Obscura addresses this at the source. As Beldex adoption grows, proof data volume grows much more slowly than under prior proof systems. Full node sync times remain manageable at scale.
Foundation for BeldexAI sidechain
Efficiency improvements in Obscura prepare the base chain for the planned BeldexAI sidechain - a decentralized, privacy-focused platform for AI model training and inference. The additional transaction volume expected from AI computation workloads requires the proof efficiency that Obscura delivers before the sidechain launch is economically viable.
