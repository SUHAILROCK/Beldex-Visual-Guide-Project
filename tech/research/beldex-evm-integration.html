The Hybrid Architecture

Beldex EVM uses a layered architecture that combines UTXO privacy with account-based smart contracts. Each layer serves a distinct purpose:

??

Layer 1: Main Chain (UTXO) LIVE

Purpose: Fast, private financial transactions
Features: RingCT, Bulletproof++, Privacy by Default
Consensus: Proof of Stake (11+1 nodes, VRF planned)

??

Layer 2: EVM Execution PoC

Purpose: Smart contracts, DeFi, dApps
Features: Full EVM, Solidity, Ethereum Tools
Privacy: zk-SNARKs (planned), confidential amounts

Bridge Layer - The Technical Challenge: Connects L1 and L2. Users "burn" coins on the main chain to receive credit on the EVM layer.

? SOLVED

UTXO ? Account

Users burn coins on main chain, receive credit on EVM layer. Double-spending prevented via cryptographic burn proofs.

? IN RESEARCH

Account ? UTXO

Moving funds back to private main chain. Challenge: Generate valid RingCT transactions from smart contract state. Still under active research.

This hybrid approach combines the strengths of two fundamentally different models.

How Smart Contracts Work on Beldex

Proven at the Proof-of-Concept level - here is the full 6-step pipeline:

Contract Creation

Developer writes Solidity smart contract (exactly like Ethereum)

Tools: Remix, Hardhat, Truffle - all compatible

Transaction Submission

User creates Beldex transaction with contract bytecode embedded

Private transaction amounts via RingCT on Layer 1

Block Inclusion

Contract transaction added to block by PoS validator

Validated through masternode network consensus

EVM Execution

Python-based Py-EVM processes contract on all synchronized nodes

Contract code is public, execution verified by all

State Update

Contract state updated globally, subsequent transactions can trigger functions

Persistent state across the network

Privacy Preservation

Contract code is PUBLIC (anyone can verify). Transaction amounts & identities remain PRIVATE (RingCT).

Future: ZK-SNARKs for contract-level privacy

Step 6 raises a critical question: how is privacy maintained when contract code must be public?

Privacy Strategy Explained

Two approaches were considered for adding privacy to smart contracts. Only one was viable:

? Rejected

Encrypted Smart Contracts

Idea:
Encrypt the smart contract code itself
Critical Problem:
Validators cannot execute or verify encrypted code
Why It Fails:
Decentralized networks require public verification for consensus
Fundamentally impossible ?
? Chosen

Layer-Based Privacy

Layer 1 (Main Chain):
RingCT hides amounts, senders, and receivers
Smart Contract Layer:
Code is public, but transaction data remains private
Future Enhancement:
ZK-SNARKs add privacy to contract inputs/outputs
Transparent + Private ?

Value locked by a contract could be used twice without a fix

? PoC Proven
Problem

Value Reuse Risk

Solution (PoC)

Cryptographic Linking

  • Approach: Protocol-level consensus rules
  • How: Unique state transitions linked
  • Benefit: Reuse prevented + privacy maintained
